## 4. Gap Analysis

**GAP-1: No File Upload Capability** -- Severity: CRITICAL
The Knowledge Vault cannot accept user-uploaded files. The investor documents page shows upload UI (drag-drop, file type labels) but it is entirely non-functional with hardcoded mock data. The interview documents page has a shell upload area with no backend. This is the single largest gap -- every benchmarked competitor supports file upload.

**GAP-2: Visibility Not Enforced** -- Severity: HIGH
The `visibility` field (`org`, `owner_only`, `advisor_only`) exists in the schema and is set on creation, but `listVaultItems()` does not filter by visibility. All org members see all items regardless of the visibility setting. This is a data exposure risk.

**GAP-3: Full-Text Search Index Unused** -- Severity: MEDIUM
Migration 010 creates a GIN tsvector index on `title || description`, but the actual query uses `ilike` substring matching. The full-text index provides ranking, stemming, and performance benefits that are going unused.

**GAP-4: No Content Renderer** -- Severity: MEDIUM
Vault items store content as JSONB but the UI only displays metadata (title, type, version, provenance). Users cannot view the actual content of any vault item from within the vault browser. This makes the vault a metadata browser rather than a document viewer.

**GAP-5: AI Outputs Page Not Wired** -- Severity: LOW
The `/vault/ai-outputs` sub-page shows only an empty state. It should filter vault items to show only AI-generated types (ai_analysis, narrative, scenario_output) but instead is a static placeholder.

**GAP-6: Guides Not Seeded** -- Severity: LOW
`vault-starter-content.ts` defines 5 starter guides but they are never inserted into the database. The guides page displays hardcoded data that duplicates the starter content definitions rather than querying the vault.

**GAP-7: Investor Document Vault Disconnected** -- Severity: HIGH
`/investors/documents/page.tsx` is a completely separate mock implementation with its own hardcoded document list, category system, and access log. It shares no infrastructure with the Knowledge Vault. This creates two parallel "document" concepts that should be unified.

**GAP-8: No Download/Export from Vault** -- Severity: MEDIUM
The access log supports `downloaded` and `exported` events, and the vault detail panel has no download or export buttons. Users cannot extract content from the vault.

**GAP-9: No Approval Workflow** -- Severity: MEDIUM
Status transitions (draft -> final -> superseded -> archived) have no approval gates. Any advisor can change status directly. Competitors like Vena and Planful require review/approval before finalisation.

**GAP-10: Access Log Not Surfaced in UI** -- Severity: LOW
The `vault_access_log` table records all access events but there is no UI to view this data. The investor documents page shows a mock access log, suggesting this is a desired feature.

## 5. Prioritised Recommendations

### P0 -- Must Build (Blocking or Security)

**P0-1: File Upload Pipeline**
Add Supabase Storage bucket for vault files. Create `/api/vault/[orgId]/upload` endpoint accepting multipart form-data. Support PDF, DOCX, XLSX, PPTX. Wire drag-drop UI on investor documents page and interview documents page. Store file reference in vault_items alongside JSONB content.

**P0-2: Enforce Visibility Filtering**
Add visibility-aware filtering to `listVaultItems()`. When querying, check `visibility` against the requesting user's role. `owner_only` items should only be visible to the `created_by` user and admins. `advisor_only` items should exclude viewer-role users. This is a data access control fix.

**P0-3: Fix Search to Use Full-Text Index**
Replace `ilike` search with `to_tsvector('english', title || ' ' || description) @@ plainto_tsquery('english', ?)` to leverage the existing GIN index. This gives ranked results, stemming, and better performance.

### P1 -- Should Build (Competitive Parity)

**P1-1: Content Renderer for Vault Items**
Build a JSONB content viewer that renders vault item content appropriately by type -- board packs as structured reports, KPI snapshots as metric cards, narratives as formatted text, etc.

**P1-2: Wire AI Outputs Sub-Page**
Replace the static empty-state page with a filtered view of the main vault, showing only `ai_analysis`, `narrative`, `scenario_output`, and `anomaly_report` types. Reuse `VaultBrowserClient` with a pre-set type filter.

**P1-3: AI Document Analysis Pipeline**
Build text extraction from uploaded PDFs/DOCX. Send extracted text to Claude API for summarisation, entity extraction, and key metric identification. Store AI analysis as a linked vault item.

**P1-4: Version Diff Viewer**
Build side-by-side or inline diff view for comparing vault versions. For JSONB content, implement structural diff highlighting. Critical for governance use case.

**P1-5: Unify Investor Documents with Knowledge Vault**
Migrate `/investors/documents` from hardcoded mock to a filtered view of the Knowledge Vault. Add `investor_document` to the vault_item_type enum. Add investor-specific sharing controls (secure links with expiry).

**P1-6: External Sharing with Secure Links**
Build shareable links for vault items with expiry dates, view-only mode, and access tracking. Essential for the investor portal use case.

**P1-7: Download/Export from Vault**
Add download buttons to vault detail panel. For JSONB content, offer PDF and JSON export. For stored files, offer direct download. Log all downloads to vault_access_log.

### P2 -- Nice to Have (Differentiation)

**P2-1: Template Engine**
Build template creation and management. Seed starter content on org creation. Allow templates with dynamic data bindings (e.g., "insert latest revenue KPI here").

**P2-2: Audit Trail Viewer**
Surface vault_access_log in the UI. Show who viewed, downloaded, or shared each item. Add filtering by date, user, and action type. Add CSV export for compliance.

**P2-3: Tag Management UI**
Add tag autocomplete, bulk tagging, and tag-based filtering to the vault browser. Show tag usage statistics.

**P2-4: Approval Workflows**
Add review/approval gates for status transitions. Require a second advisor to approve `draft -> final`. Notify stakeholders on status changes.

**P2-5: Semantic Search**
Generate embeddings for vault content using Claude/OpenAI. Store in pgvector. Enable natural language queries across the vault.

## 6. Architectural Concerns

**ARCH-1: Visibility Not Enforced at Query Level**
Severity: HIGH. The `visibility` column on `vault_items` is set on insert but never checked during `listVaultItems()`. The RLS policy only checks `org_id`, not `visibility`. This means `owner_only` documents are visible to all org members. Fix: Either add visibility filtering in the application query OR add RLS policies that check visibility against the requesting user's role and ID.

**ARCH-2: Dual Document Systems**
Severity: MEDIUM. The investor documents page (`/investors/documents`) operates as a completely separate document management system with its own types, categories, access log, and sharing model. It shares zero code with the Knowledge Vault. This will create maintenance burden and user confusion. Fix: Unify under the vault infrastructure with investor-specific item types and sharing features.

**ARCH-3: JSONB Content Without Renderer**
Severity: MEDIUM. All vault content is stored as JSONB but there is no mechanism to render or display it. The vault is effectively a metadata catalogue without a content viewer. Each item type (board_pack, kpi_snapshot, etc.) will need a type-specific renderer. Plan this before the item count grows.

**ARCH-4: No Binary File Storage Integration**
Severity: HIGH. The vault schema and API only support JSONB content. There is no Supabase Storage bucket, no file reference field, and no file upload endpoint. Adding binary file support will require a schema change (new `file_path` or `storage_key` column) and a storage bucket with its own RLS policies.

**ARCH-5: Search Uses ilike Instead of tsvector**
Severity: LOW. The database has a GIN full-text search index that the application does not use. The `ilike` approach will degrade in performance as the vault grows and does not support relevance ranking. This is a straightforward fix in `listVaultItems()`.

**ARCH-6: Auto-Store Uses Fire-and-Forget Pattern**
Severity: LOW. `autoStoreToVault` catches all errors silently. If vault storage fails, the user has no indication that their output was not preserved. Consider adding a retry mechanism or surfacing failures as notifications.
